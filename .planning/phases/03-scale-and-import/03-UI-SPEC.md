## Spacing Scale

Inherited from Phase 2 contract — no changes. All values are multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, stepper button icon padding, URL badge gap |
| sm | 8px | Gap between stepper button and number display, compact element spacing |
| md | 16px | Default element spacing — between form fields, inner card padding |
| lg | 24px | Section padding — between constraint panel sections |
| xl | 32px | Layout gaps — between form card and result card |
| 2xl | 48px | Major section breaks — not used in Phase 3 |
| 3xl | 64px | Page-level spacing — not used in Phase 3 |

Exceptions (Phase 3 additions):
- Stepper buttons (+/−): minimum 36px height, 36px width touch target. Use `min-w-9 h-9` (36px) to match the pill touch target floor from Phase 2.
- URL detection badge: `px-2 py-1` (8px/4px), matching TagInput chip spacing.

Inherited from Phase 2 contract — no new type roles added.

Phase 3 typography notes:

- Error message for scraping failure: 14px weight 400, `text-destructive` — same style as Phase 2 transformation error.

### Servings field placement in form (D-01, D-03)

The `<section> Servings` block appears **between the constraint panel and the submit button** — after dietary needs, before the submit action. This groups data-entry above the action. Label is "Servings". ServingStepper is left-aligned (not full-width).

### ServingStepper anatomy

```
[−] [N] [+]
```

- Three adjacent elements with a shared visual border group: `flex items-center`
- Decrement button: `min-w-9 h-9 flex items-center justify-center border border-border bg-background hover:bg-muted rounded-l-md`
- Number display: `min-w-12 h-9 flex items-center justify-center border-y border-border bg-background text-sm font-normal text-center`
- Increment button: `min-w-9 h-9 flex items-center justify-center border border-border bg-background hover:bg-muted rounded-r-md`
- Decrement disabled when `value <= min` — use `disabled:opacity-40 disabled:cursor-not-allowed`
- No accent color — stepper uses border/muted tokens only

### Result card serving stepper (D-02, D-04)

In `RecipeCard`, the existing `<p className="text-sm text-muted-foreground">Serves {recipe.servings}</p>` is replaced with:

```
<div className="flex items-center gap-3 mt-1">
  <ServingStepper value={currentServings} onChange={setCurrentServings} min={1} />
  <span className="text-sm text-muted-foreground">(original: {recipe.servings})</span>
</div>
```

- `currentServings` is local state in RecipeCard, initialized to `recipe.servings` on mount.
- `onChange` triggers client-side quantity rescaling — no backend call.
- "(original: N)" reads `recipe.servings` directly — the LLM-inferred value from `TransformedRecipe.servings`.

### URL detection badge placement (D-06)

The badge appears **inside a relative-positioned wrapper div** that wraps the textarea. Position: `absolute top-2 right-2`. This places it in the top-right corner of the textarea, over the textarea (but the textarea has `pt-8` or equivalent when badge is visible to prevent text overlap in the first line — OR the badge floats above the text as an overlay without text adjustment, acceptable given the badge is small). Recommended: overlay approach — the badge is small enough (≈ "🔗 URL detected" at 12px) that it does not obscure meaningful content at the top-right of a 10-row textarea.

Badge visibility rule: rendered when `recipeText.startsWith("http://") || recipeText.startsWith("https://")`. Computed on every `onChange` event. No debounce needed.

## Interaction Contract

### ServingStepper (D-03)

- Decrement (`−`): decrements `value` by 1. Disabled when `value <= min`. `type="button"` to prevent form submission.
- Increment (`+`): increments `value` by 1. No upper bound (unclamped at top). `type="button"`.
- Number display: not editable directly (display only). Do not use `<input type="number">` — the +/− buttons are the exclusive control.
- Min clamp: enforced at the button level (decrement disabled at min) AND in `onChange` handler (`Math.max(min, newValue)`).
- Keyboard: decrement/increment buttons receive normal focus and respond to Space/Enter (default button behavior — no extra handling required).
- ARIA: decrement button `aria-label="Decrease servings"`, increment button `aria-label="Increase servings"`, number display `aria-live="polite"` so screen readers announce value changes.

### UrlDetectionBadge (D-05, D-06)

- Appears: when `recipeText.startsWith("http://") || recipeText.startsWith("https://")`.
- Disappears: when the leading characters no longer form a URL prefix.
- No animation on appear/disappear — instant conditional render.
- Not interactive — no click handler. Read-only feedback element.
- `aria-hidden="true"` — the badge is visual confirmation only; screen readers do not need to announce it (the form submit will behave correctly regardless).

### Scraping failure error (D-07, D-08)

- Triggered when: backend returns a non-2xx response specifically from the URL scraping path (distinct from LLM transformation failure).
- Frontend distinguishes by: response body or status code from backend (e.g., 422 with a specific error type, or any non-200 when input was URL-detected).
- The same error state variable (`error`) used in Phase 2 is reused — no new state variable.
- Error clears: at top of next `handleSubmit` call, same as Phase 2.
- URL stays in the textarea: `recipeText` state is NOT cleared on scraping failure. The user sees the URL still present.
- Form stays visible: `formCollapsed` remains `false` on error, identical to Phase 2 behavior.

### Form servings field default value

- Default value when the form first loads: `2` (a common recipe serving count — not 1, which implies single-serving edge case).
- If the user has not changed it, the value `2` is sent to the backend as `targetServings`.
- The backend applies `ScalingService.scale(recipe, targetServings)` after the LLM call returns `recipe.servings` (the LLM-inferred count). If `targetServings == recipe.servings`, the scaled result is mathematically identical to the LLM output.

### Post-result stepper behavior

- Initialized: `currentServings = recipe.servings` (LLM-inferred value).
- On stepper change: client-side rescaling updates the displayed quantities without a backend call.
- The form's `targetServings` field and the result card stepper are independent — adjusting the result card stepper does NOT update the form field. If the user clicks "Edit / start over", the form field retains whatever value it had before submission.

## Copywriting Contract

Inherited from Phase 2 — all Phase 2 copy unchanged. Phase 3 additions:

| Element | Copy |
|---------|------|
| Servings field label | Servings |
| Servings stepper decrement aria-label | Decrease servings |
| Servings stepper increment aria-label | Increase servings |
| URL detection badge | URL detected |
| Result card: current servings annotation | (original: {N}) |
| Result card: servings region (replaces "Serves N") | (rendered via ServingStepper + "(original: N)" span — no static label) |
| Scraping error message | Couldn't import this URL. Open it, copy the recipe text, and paste it here. |
| Scraping error prefix glyph | ⚠ (Unicode warning sign, same as Phase 2 error treatment) |
| Full scraping error rendered | ⚠ Couldn't import this URL. Open it, copy the recipe text, and paste it here. |

Copywriting rationale:
- "Servings" (not "Target servings" or "Number of servings") — shortest unambiguous label.
- "URL detected" (not "Importing from URL" or "Web import") — present-tense confirmation, not a promise. The actual scraping has not happened yet.
- Scraping error copy (from CONTEXT.md specifics): "Couldn't import this URL. Open it, copy the recipe text, and paste it here." — includes the recovery path inline, per D-07 and the specific note in the `<specifics>` block.
- "(original: N)" — parenthetical to visually subordinate it relative to the stepper which shows the active value.

Destructive actions in Phase 3: none. "Edit / start over" behavior is identical to Phase 2 (stateless, no confirmation required).
