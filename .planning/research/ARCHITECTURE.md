**Researched:** 2026-04-19
**Confidence:** HIGH

## System Overview

DietCode is a single-user, stateless request-response system. No sessions, no persistence, no shared state.

```
Browser (React + Vite)
       |
       | REST + SSE (text/event-stream)
       |
Spring Boot (Kotlin) — single process
  ├── RecipeController         HTTP entry points
  ├── RecipeIngestionService   Parse text OR scrape URL → RecipeDocument
  ├── TransformationService    Build prompt + call LLM → TransformedRecipe (SSE)
  ├── ScalingService           Adjust quantities (pure arithmetic, no LLM)
  └── ExportService            Serve print-ready payload to frontend

       |                            |
  Spring AI (ChatClient)       Jsoup (HTTP + HTML parse)
       |                            |
  LLM Provider API            Recipe site HTML / JSON-LD
  (Anthropic Claude Haiku 3.5)
```

---

## Component Boundaries

| Component | Responsibility | Key Rule |
|-----------|---------------|----------|
| `RecipeController` | HTTP routing, SSE setup, request validation | Routes only; no business logic |
| `RecipeIngestionService` | Accept raw text or fetch+parse URL → `RecipeDocument` | Never calls LLM |
| `TransformationService` | Build prompt; call Spring AI `ChatClient`; parse structured response | Never scrapes |
| `ScalingService` | Multiply/divide quantities by serving ratio | Pure arithmetic; no external calls |
| `ExportService` | Serialize `TransformedRecipe` for frontend | Backend stays stateless; no PDF generation |

Each service is independently unit-testable.

## Canonical Data Types

```kotlin
data class RecipeDocument(
    val title: String,
    val ingredients: List<String>,    // raw strings as found
    val instructions: List<String>,   // ordered steps
    val servings: Int?,               // original count, nullable
    val sourceUrl: String?
)

data class TransformedRecipe(
    val title: String,
    val diet: String,
    val intolerances: List<String>,
    val ingredients: List<IngredientLine>,  // structured for scaling
    val instructions: List<String>,
    val substitutionNotes: List<String>,    // LLM-generated explanations
    val servings: Int
)

data class IngredientLine(
    val quantity: Double?,     // null = "to taste", "a pinch", etc.
    val unit: String?,
    val ingredient: String,
    val preparation: String?   // "finely chopped", "room temperature"
)
```

**Critical:** `IngredientLine` must be structured — not a free-form string — because `ScalingService` needs numeric `quantity`. The LLM must return ingredients in this parsed form via Spring AI's `BeanOutputConverter`.

## Data Flow

**Path A — Pasted Text:**
```
POST /api/recipe/transform { text, diet, intolerances, servings }
  → RecipeIngestionService.parseText(text) → RecipeDocument
  → TransformationService.transform(doc, diet, intolerances) → SSE stream
  → [stream closes] → ScalingService.scale(result, targetServings) → TransformedRecipe
  → Final JSON to frontend
```

**Path B — URL Import (three-tier extraction):**
```
POST /api/recipe/transform { url, diet, intolerances, servings }
  → RecipeIngestionService.fetchUrl(url)
      Tier 1: JSON-LD <script type="application/ld+json"> with @type=Recipe
              → parse schema.org fields (recipeIngredient, recipeInstructions, recipeYield)
      Tier 2: Microdata fallback — [itemprop=recipeIngredient]
      Tier 3: Heuristic — <ul>/<ol> inside article/main/[class*=recipe] containers
      Tier 4: Error → prompt user to paste text manually
  → RecipeDocument
  → [same as Path A from here]
```

## LLM Prompt Design

**System prompt:**
```
You are a professional recipe developer who adapts recipes to meet dietary needs.
Rewrite the recipe with required ingredient substitutions.
Rules:
- Preserve the dish's character, technique, and flavor profile as much as possible
- Make ONLY substitutions required by the dietary constraints
- For each substitution, include a brief note explaining why
- Return ONLY valid JSON matching the schema provided — no markdown fences, no commentary
```

**Temperature: 0.3** — recipe substitution is deterministic reasoning, not creative generation. High temperature produces inconsistent quantities.

**User message template:**
```
Adapt this recipe for: {diet}
Remove/replace these ingredients: {intolerances}

Original recipe:
{recipeText}

Return JSON matching this schema:
{jsonSchema}
```

The JSON schema is auto-generated by Spring AI's `BeanOutputConverter` via `converter.getFormat()`.

## Streaming: SSE via SseEmitter

Recipe transformations take 5–20 seconds. Use SSE streaming for immediate visual feedback. **WebFlux is NOT required** — `SseEmitter` + `ExecutorService` on the servlet stack is sufficient for one user.

```kotlin
@GetMapping("/api/recipe/transform/stream", produces = [MediaType.TEXT_EVENT_STREAM_VALUE])
fun transformStream(@RequestBody request: TransformRequest): SseEmitter {
    val emitter = SseEmitter(60_000L)
    executor.submit {
        try {
            chatClient.prompt()
                .system(systemPrompt)
                .user(buildUserMessage(request))
                .stream()
                .content()
                .doOnNext { chunk -> emitter.send(SseEmitter.event().data(chunk)) }
                .doOnComplete { emitter.complete() }
                .doOnError { e -> emitter.completeWithError(e) }
                .subscribe()
        } catch (e: Exception) {
            emitter.completeWithError(e)
        }
    }
    return emitter
}
```

## API Surface

| Method | Path | Request | Response |
|--------|------|---------|----------|
| `POST` | `/api/recipe/ingest` | `{ url?, text? }` | `RecipeDocument` |
| `POST` | `/api/recipe/transform/stream` | `{ recipe, diet, intolerances, servings }` | SSE stream |
| `POST` | `/api/recipe/scale` | `{ recipe: TransformedRecipe, targetServings }` | `TransformedRecipe` |

`/api/recipe/scale` allows re-scaling without paying another LLM call.

CORS: configure `WebMvcConfigurer` to allow `http://localhost:5173` for dev.

## Suggested Build Order

1. **Core Data Pipeline (no LLM):** `RecipeIngestionService` + data models — fully testable without API key
2. **LLM Integration (synchronous):** Wire Spring AI `ChatClient` with `BeanOutputConverter`; get the prompt right before streaming
3. **Scaling Logic:** Pure arithmetic on `TransformedRecipe`; trivially decoupled
4. **Streaming UX:** Replace synchronous call with `SseEmitter` + frontend `EventSource` — additive, not a rewrite
5. **Export:** Print CSS + optional Google Drive — no backend changes

---

## Anti-Patterns to Avoid

- **Scaling inside the LLM prompt** — LLM will hallucinate quantities; scaling is arithmetic, keep it deterministic
- **Free-form ingredient strings from LLM** — makes scaling impossible; require `IngredientLine` structured output
- **WebFlux migration for streaming** — `SseEmitter` is sufficient for one user
- **Skipping JSON-LD check** — most major recipe sites embed it; always check first
- **Scaling logic in frontend** — parsed `IngredientLine` data lives on backend; call `/api/recipe/scale` instead
